# API Design Audit Report

**Date:** 2026-03-25
**Scope:** ~120 API endpoints across 30 handler files
**Auditor:** Claude Code (automated, 4 parallel agents)

## Executive Summary
- API design health: **NEEDS ATTENTION**
- Findings: 4 critical, 11 high, 18 medium, 9 low — 42 total (deduplicated to 35 unique)
- Consistency score: **6/10**
- Complete CRUD: 5 of 28 resources (Projects, Workspaces, Webhooks, Feature Flags, Branch Protection)

Key themes: (1) DELETE endpoints use inconsistent status codes (200 vs 204), (2) comment deletion missing everywhere, (3) many resources lack GET-single endpoints, (4) validation errors split between 400 and 422.

## Response Format Consistency

| Pattern | Consistent? | Violations |
|---|---|---|
| List → ListResponse{items,total} | **No** | API1: sessions children/iframes return bare Vec; API2: workspace members bare Vec |
| Create → 201 + resource | **Yes** | — |
| Delete → 204 | **No** | API3: ~15 DELETE endpoints return 200 `{"ok":true}`, some return 204 |
| Error → JSON `{"error":...}` | **Yes** | — |
| Validation → 422 with fields | **No** | API4: most validation returns 400 BadRequest, not 422 Validation |
| DateTime format | **No** | API5: workspaces use `.to_rfc3339()` String, others use `DateTime<Utc>` |
| Pagination (limit/offset) | **No** | API6: releases list, workspace members, session children not paginated |

## Findings

### CRITICAL

**API1: Inconsistent DELETE response codes across entire API**
~15 DELETE endpoints return `200 OK` with `{"ok": true}` while workspaces/members return `204 No Content`. Both patterns exist simultaneously, making the API unpredictable for clients.
- Affected: projects, webhooks, secrets, delegations, roles, tokens, ssh-keys, gpg-keys, passkeys, flags, branch-protection, service-accounts
- Fix: Standardize ALL DELETE endpoints to return `204 No Content` with empty body

**API2: Issue deletion endpoint missing**
Issues can be created, listed, updated, but never deleted. No soft-delete either. Stale issues accumulate with no cleanup path.
- Fix: Add `DELETE /api/projects/{id}/issues/{number}` with soft-delete pattern

**API3: MR deletion endpoint missing**
Same as API2 for merge requests. Abandoned/test MRs cannot be removed.
- Fix: Add `DELETE /api/projects/{id}/merge-requests/{number}` with soft-delete

**API4: Permission denial status codes inconsistent**
Read permission denials correctly return 404 (per CLAUDE.md), but write permission denials inconsistently return either 403 or 404 depending on the handler. Some write-operation helpers (`require_agent_run` in sessions.rs) return 404 instead of 403.
- Fix: Enforce rule: `require_X_read()` → 404, `require_X_write()` → 403, `require_admin()` → 403

### HIGH

**API5: Comment deletion missing everywhere**
Issue comments and MR comments can be created and updated, but never deleted. No GET-single endpoint for individual comments either.
- Fix: Add `DELETE` and `GET` for `/comments/{comment_id}` on both issues and MRs

**API6: Bare Vec responses instead of ListResponse envelope**
`GET /api/projects/{id}/sessions/{sid}/children` and `GET /api/projects/{id}/sessions/{sid}/iframes` return raw `Json<Vec<T>>` instead of `ListResponse<T> { items, total }`.
- Fix: Wrap in `ListResponse<T>` to match all other list endpoints

**API7: Workspace members list returns bare Vec**
`GET /api/workspaces/{id}/members` returns `Vec<MemberResponse>` without pagination envelope.
- Fix: Return `ListResponse<MemberResponse>` with pagination params

**API8: Validation errors use BadRequest (400) instead of Validation (422)**
The `ApiError::Validation` variant exists with structured `{"error":"validation error","fields":[...]}` at 422, but most validators in `src/validation.rs` return `ApiError::BadRequest` at 400 instead. Clients can't distinguish "bad input format" from "valid format, business rule violation."
- Fix: Migrate validation helpers to return `ApiError::Validation(vec![...])` with field names

**API9: MR reviews missing GET-single, UPDATE, DELETE**
Reviews can only be created and listed. No way to retrieve, update, or retract a review.
- Fix: Add `GET /reviews/{review_id}`, consider `PATCH` for verdict updates and `DELETE` for retraction

**API10: Roles missing GET-single, UPDATE, DELETE**
Admin roles can be created and listed but not retrieved individually, renamed, or deleted.
- Fix: Add `GET /api/admin/roles/{id}`, `PATCH /api/admin/roles/{id}`, `DELETE /api/admin/roles/{id}`

**API11: Generic `{"ok": true}` responses instead of typed structs**
Multiple endpoints return `Json<serde_json::Value>` with `{"ok": true}`. No TypeScript type generation possible, poor API documentation.
- Affected: session message/stop, all `{"ok": true}` delete responses
- Fix: Either return `204 No Content` (for deletes) or define typed response struct

**API12: `/api/users/list` redundant naming**
List endpoint uses `/api/users/list` instead of the standard `GET /api/users` pattern used by all other resources.
- Fix: Change to `GET /api/users` (may require separating from `POST /api/users` if not already)

**API13: Releases list endpoint lacks pagination**
`GET /api/projects/{id}/releases` returns all releases without `limit`/`offset` support, unlike every other list endpoint.
- Fix: Add `ListParams` query extraction with default limit=50, max=100

**API14: Parse errors mapped to Internal (500) instead of BadRequest (400)**
`users.rs` `parse_user_type()` maps parse failures to `ApiError::Internal`, making client input errors appear as server errors.
- Fix: Map parse failures to `ApiError::BadRequest("invalid user type")`

### MEDIUM

**API15: DateTime serialization inconsistent in workspaces**
`WorkspaceResponse` and `MemberResponse` manually convert timestamps to `String` via `.to_rfc3339()`, while all other response types use `DateTime<Utc>` directly (which also serializes to ISO 8601).
- Fix: Change workspace response `created_at` fields from `String` to `DateTime<Utc>`

**API16: Release assets missing list, update, delete**
Can upload and download release assets, but cannot list assets independently, update metadata (rename), or delete individual assets.
- Fix: Add `GET /assets`, `PATCH /assets/{aid}`, `DELETE /assets/{aid}` under releases

**API17: Session DELETE endpoint missing**
Agent sessions can be created/updated/stopped, but no DELETE endpoint exists. The `stop` action is POST-based; there's no way to clean up terminated sessions from the list.
- Fix: Add `DELETE /api/projects/{id}/sessions/{sid}` or consider session stop as sufficient

**API18: Deploy targets missing UPDATE, DELETE**
Targets can be created and read but not updated or deleted.
- Fix: Add `PATCH /api/projects/{id}/targets/{tid}` and `DELETE /api/projects/{id}/targets/{tid}`

**API19: Delegations missing GET-single, UPDATE**
Cannot retrieve delegation details individually or extend expiry.
- Fix: Add `GET /api/admin/delegations/{id}` and `PATCH /api/admin/delegations/{id}`

**API20: Service accounts missing GET-single, UPDATE**
Cannot retrieve or update individual service accounts.
- Fix: Add `GET /api/admin/service-accounts/{id}`, `PATCH /api/admin/service-accounts/{id}`

**API21: Workspace members missing UPDATE (role change)**
Cannot change a member's role without remove + re-add.
- Fix: Add `PATCH /api/workspaces/{id}/members/{mid}` with `{"role": "admin"}`

**API22: SSH keys missing GET-single**
Cannot retrieve metadata (fingerprint, algorithm) for a single key.
- Fix: Add `GET /api/users/me/ssh-keys/{id}`

**API23: Commands missing GET-single**
Cannot retrieve a single command definition.
- Fix: Add `GET /api/commands/{id}`

**API24: Inconsistent 404 message specificity**
NotFound errors vary between `"project"`, `"secret"`, `"session"`, `"session namespace"`. Resource type names in 404 messages could leak internal information.
- Fix: Standardize to generic `"not found"` for private resources, keep resource names only for public-facing 404s

**API25: Secret requests missing reject/cancel flow**
Requests can only be created and completed. No way to reject or cancel a pending request.
- Fix: Add `POST /api/projects/{id}/secret-requests/{rid}/reject` and `DELETE .../{rid}` for cancellation

**API26: Conflict error messages inconsistent granularity**
Some unique constraint violations give specific messages (`"a project named 'X' already exists"`) while the generic `From<sqlx::Error>` returns `"resource already exists"`.
- Fix: Add handler-specific constraint detection for all resources that can conflict

**API27: `PATCH /api/notifications/{id}/read` uses verb in URL**
The `read` segment is a verb. Should be a property update on the notification itself.
- Fix: Change to `PATCH /api/notifications/{id}` with body `{"read": true}`, or keep as action endpoint with `POST /api/notifications/{id}/mark-read`

**API28: LLM providers, passkeys, provider-keys missing GET-single**
Multiple user-scoped resources lack individual retrieval endpoints.
- Fix: Add `GET` single endpoints for each

**API29: Dynamic sqlx queries in some handlers lack specific error handling**
`releases.rs`, `deployments.rs`, `branch_protection.rs` use `sqlx::query()` (dynamic) and rely on blanket `From<sqlx::Error>` which may not catch unique violations as 409.
- Fix: Add explicit error matching for constraint violations in dynamic query handlers

**API30: Global/workspace secrets missing GET-single**
Can list and delete global/workspace secrets, but cannot read a single secret's metadata.
- Fix: Add `GET /api/admin/secrets/{name}` and `GET /api/workspaces/{wid}/secrets/{name}`

**API31: API tokens missing GET-single**
Cannot retrieve metadata for a single API token.
- Fix: Add `GET /api/tokens/{id}`

**API32: Inconsistent list param struct naming**
List parameter structs vary: `ListProjectsParams`, `ListIssuesParams`, `ListMrParams`, generic `ListParams`, `ListPipelinesParams`, `ListSessionsParams`, `ListCommandsParams`.
- Fix: Standardize on `List{Resource}Params` naming convention across all endpoints

### LOW

**API33: Passkey route prefix inconsistency**
Registration uses `/api/auth/passkeys/register/begin` (plural) while login uses `/api/auth/passkey/login/begin` (singular).
- Fix: Standardize to `/api/auth/passkeys/...` (plural) everywhere

**API34: No sort parameter on any list endpoint**
All lists default to `ORDER BY created_at DESC` with no client-controllable sort.
- Fix: Add optional `sort` and `direction` query params to list endpoints

**API35: No API versioning**
No `/v1/` prefix, no version headers, no deprecation mechanism.
- Fix: Establish versioning strategy before breaking changes. Consider header-based (`API-Version: 2026-03-25`)
